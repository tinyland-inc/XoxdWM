# XR Kernel Boot Failure — Root Cause Analysis

## Final Diagnosis (2026-03-16)

**Root cause: missing `root=` parameter in the BLS boot entry.**

The manually-created BLS entry for the XR kernel omitted `root=/dev/mapper/rl00-root`.
Without this parameter, dracut's initrd activates the LVM volumes (`rl00/root`, `rl00/swap`)
and they appear in `/dev/mapper/`, but systemd has no instruction to mount the root
filesystem to `/sysroot`. The `initrd-switch-root.service` then fails because
`/sysroot/etc/os-release` does not exist (the directory is empty).

```
Failed to switch root: Specified switch root path '/sysroot' does not seem to be an OS tree. os-release file is missing.
```

The ELRepo kernel's BLS entry (auto-generated by `kernel-install`) included `root=/dev/mapper/rl00-root`.
The XR kernel's entry (manually created after `%posttrans` failed) did not.

## What the Debug Capture Showed

A custom dracut emergency hook (`99debug-capture`) wrote state to `/boot/xr-boot-debug.txt`
at the point of failure. Key findings:

| Component | State | Expected |
|-----------|-------|----------|
| NVMe drives | Detected (nvme0n1p1, nvme1n1p1) | Correct |
| LVM PVs | All 3 active (rl00, data, hdd) | Correct |
| LVM LVs | rl00/root active, /dev/mapper/rl00-root exists | Correct |
| /sysroot | Empty (only `var/` from rpc_pipefs) | Should have full OS tree |
| mount of rl00-root | **Never happened** | Should mount to /sysroot |
| initrd-switch-root | "os-release file is missing" | Should find /sysroot/etc/os-release |

## The Fix

Add `root=/dev/mapper/rl00-root` to the BLS entry options line:

```diff
- options ro crashkernel=2G-64G:256M,64G-:512M resume=UUID=... rd.lvm.lv=rl00/root rd.lvm.lv=rl00/swap
+ options root=/dev/mapper/rl00-root ro crashkernel=2G-64G:256M,64G-:512M resume=UUID=... rd.lvm.lv=rl00/root rd.lvm.lv=rl00/swap
```

## Red Herrings Investigated

These were all investigated but were NOT the cause:

| Investigation | Finding | Impact |
|---------------|---------|--------|
| CONFIG_FW_LOADER_USER_HELPER | ELRepo boots fine WITH it enabled | None (was never the cause) |
| CONFIG_DELL_RBU selecting FW_LOADER_USER_HELPER | Kconfig dependency confirmed, but irrelevant | None |
| CONFIG_DEBUG_INFO_BTF / DWARF5 | Both kernels have BTF working | None |
| Stale initramfs (missing NVMe modules) | Fixed, but not the boot failure cause | Required fix, but not root cause |
| XR patches (DSC + EDID) | QEMU boot proved kernel is fine | None |
| GCC_PLUGINS config difference | ELRepo has it, XR doesn't — no effect | None |

## Why QEMU Didn't Catch It

QEMU tests used `root=/dev/sda` (explicitly provided), so the `root=` parameter was always present.
The QEMU test correctly showed the kernel binary boots fine — the issue was never in the kernel.

## Timeline

1. **xr2** (2026-03-14): Failed — attributed to BTF disabled (CONFIG_DEBUG_INFO_NONE)
2. **xr3**: Failed — same attribution
3. **xr4** (2026-03-15): Failed — BTF fixed but `root=` was already missing in manual BLS entry
4. **xr4 + rebuilt initramfs** (2026-03-16): Failed — NVMe modules added, but `root=` still missing
5. **xr5** (2026-03-16): Failed — FW_LOADER_USER_HELPER fixed, DELL_RBU removed, but `root=` still missing
6. **xr5 + amdgpu blacklisted**: Failed — confirmed patches aren't the issue
7. **xr5 + debug-capture hook**: Revealed `/sysroot` empty, LVM active but root not mounted
8. **xr5 + `root=` added**: **BOOTS SUCCESSFULLY**

All previous failures (xr2-xr5) may have had the same root cause. The BTF and FW_LOADER fixes
were still correct improvements, but they were not required for boot.

## Prevention

1. **Never manually create BLS entries** — use `kernel-install add` which generates correct entries
2. **The Dhall boot pipeline** (`packaging/dhall/defaults/`) always includes `root=` in the options
3. **The `boot-apply` script** renders from Dhall, ensuring all required parameters are present
4. **The `boot-test-kernel` justfile target** uses `grub2-reboot` for safe one-shot testing
5. **The debug-capture dracut hook** should be kept for future boot debugging

## Debug Capture Hook

The `99debug-capture` dracut module (`/usr/lib/dracut/modules.d/99debug-capture/`) fires on
emergency mode entry and writes diagnostic state to `/boot/xr-boot-debug.txt` by explicitly
mounting the real `/boot` partition. This is now a permanent diagnostic tool on honey.
