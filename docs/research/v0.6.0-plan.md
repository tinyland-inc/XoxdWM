# EWWM v0.6.0 Milestone Plan: First Real VR Session on Honey

## Executive Summary

v0.5.0 built the protocol and hardware abstraction layers: 9 Wayland protocols,
Beyond 2e HID control with LinuxHidTransport, GPU power management, VR interaction
patterns, and JSON configuration. The compositor talks to the Beyond over hidraw
and the display powers on via systemd/udev.

**v0.6.0 closes the loop: real photons hitting retinas through ewwm.**

The goal is an end-to-end VR session on the honey server (Dell T7810, RX 9070 XT,
Bigscreen Beyond 2e, Bigeye eye tracker) — from SteamVR bootstrap through DRM
lease to stereo rendering with eye tracking.

**Estimated total: 6-8 weeks.** Phases 1 and 2 are blocking. Phase 3 is the
milestone deliverable. Phases 4 and 5 are stretch goals.

### SteamVR Settings (for Beyond on AMD)

```json
{

  },
```

### Dependencies

- Beyond display must power on (done: HID power-on via udev trigger)
- DP-2 must be `non_desktop=1` (done: `video=DP-2:d` kernel cmdline, needs reboot)
- User in `video`, `render`, `input` groups (done)

### Risks

- **Proton HID passthrough failure** (medium): Beyond Driver may not see the HID
  device through Proton even with `PROTON_ENABLE_HIDRAW`. Mitigation: native HID
  power-on is already working — only calibration UI needs Proton.
- **Base station discovery** (low): Valve radio dongles are detected on USB.
  SteamVR should pair automatically. If not, pair via `lighthouse_console`.
- **AMD async reprojection bugs** (low): Known issue with AMD + SteamVR.
  Disable async reprojection in vrsettings.

## Phase 2: DRM Lease + Direct Display

**Goal:** ewwm compositor claims Beyond's DP-2 connector via DRM lease,
bypassing SteamVR's compositor. Monado uses DRM lease backend for direct mode.

**Effort: L (5-8 days)**

### Tasks

| # | Task | Effort | Owner |
|---|------|--------|-------|
| 2.1 | Verify `non_desktop=1` on DP-2 after reboot with kernel cmdline | S | ops |
| 2.2 | Evaluate upstream EDID quirk patch status (may be merged in 6.19+) | S | research |
| 2.3 | If not merged: add EDID quirk to NixOS kernel overlay or carry patch | M | nix |
| 2.4 | Wire `drm_lease.rs` to real DRM backend — enumerate /dev/dri/card* connectors | M | rust |
| 2.5 | Implement `wp_drm_lease_device_v1` protocol in compositor (Smithay) | L | rust |
| 2.6 | Auto-offer `non_desktop` connectors to lease clients | S | rust |
| 2.7 | Test Monado DRM lease acquisition from ewwm compositor | M | integration |
| 2.8 | Add `drm-lease-status` IPC command and wire to `ewwm-vr-display.el` | S | rust+elisp |
| 2.9 | Write fallback: detect Beyond by EDID manufacturer "BIG" if quirk missing | S | rust |

### Architecture

```
ewwm-compositor
  |
  |-- DRM master (card1)
  |     |
  |     |-- DP-1: desktop monitor (owned by compositor)
  |     |-- DP-2: Beyond 2e (non_desktop=1, offered for lease)
  |
  |-- wp_drm_lease_device_v1 (Wayland protocol)
        |
        +-- Monado requests lease for DP-2
        +-- ewwm grants lease (CRTC + connector + plane)
        +-- Monado becomes DRM master for DP-2
        +-- ewwm retains desktop rendering on DP-1
```

### Key Smithay Integration Points

`drm_lease.rs` currently has the data model. The real DRM backend wiring needs:

1. `smithay::backend::drm::DrmDevice` — obtain file descriptor
2. `smithay::wayland::drm_lease::DrmLeaseState` — protocol handler
3. Connector filtering: check `non_desktop` property via `drm::control::property::Value`
4. Lease lifecycle: `DrmLeaseBuilder::new()` -> add connector + CRTC + planes -> `build()`

### Dependencies

- Phase 1 complete (lighthouse config exported for Monado)
- Kernel `non_desktop` flag working on DP-2

### Risks

- **CRTC allocation conflict** (medium): If the kernel allocated a CRTC to DP-2
  before `non_desktop` was set, the compositor may need to release it before
  leasing. Mitigation: `video=DP-2:d` prevents initial CRTC allocation.
- **Smithay DRM lease API gaps** (medium): Smithay 0.7's `drm_lease` module
  may need upstream patches for multi-GPU or AMD-specific quirks.
  Mitigation: fall back to raw `drmModeCreateLease()` ioctl if needed.
- **Monado DRM lease client bugs** (low): Monado's DRM lease backend is
  well-tested on Intel/AMD. The Beyond is a standard DP connector.

## Phase 3: First Real VR Session

**Goal:** End-to-end: ewwm compositor renders stereo frames to Beyond via
OpenXR, with head tracking from lighthouse base stations, at 90Hz.

**This is the v0.6.0 milestone deliverable.**

**Effort: L (8-12 days)**

### Tasks

| # | Task | Effort | Owner |
|---|------|--------|-------|
| 3.1 | Wire `openxr_state.rs` to real OpenXR loader (not stub) | M | rust |
| 3.2 | Negotiate Vulkan instance/device with Monado runtime | M | rust |
| 3.3 | Create OpenXR session with `XR_KHR_vulkan_enable2` | M | rust |
| 3.4 | Implement swapchain creation — format negotiation, double/triple buffering | M | rust |
| 3.5 | Wire `vr_renderer.rs` to real swapchain images (currently test stubs) | L | rust |
| 3.6 | Stereo rendering: split 3840x1920 into 1920x1920 per eye | M | rust |
| 3.7 | Frame timing loop: `xrWaitFrame` -> `xrBeginFrame` -> render -> `xrEndFrame` | M | rust |
| 3.8 | Hit 90Hz frame budget (11.1ms) — profile and optimize | L | rust |
| 3.9 | Head tracking: read `XR_VIEW_CONFIGURATION_TYPE_PRIMARY_STEREO` poses | M | rust |
| 3.10 | Basic scene: render a single ewwm surface as a floating quad in 3D | M | rust |
| 3.11 | IPC: `vr-session-start` / `vr-session-stop` commands trigger real sessions | S | rust+elisp |
| 3.12 | Add frame timing telemetry to `ewwm-vr-scene.el` mode-line | S | elisp |

### Frame Budget Analysis

```
Target: 90Hz = 11.1ms per frame

Budget breakdown (estimated):
  xrWaitFrame + xrBeginFrame:  0.5ms
  View pose query:             0.1ms
  Scene traversal + cull:      0.3ms
  GL draw calls (per eye):     3.0ms x 2 = 6.0ms
  Texture upload (DMA-BUF):    0.5ms
  xrEndFrame + submit:         0.5ms
  ─────────────────────────────────
  Total:                       ~7.9ms (headroom: 3.2ms)
```

For Phase 3, the scene is simple (a few textured quads). The 11.1ms budget is
comfortable. Foveated rendering (Phase 4) and complex scenes will need tighter
optimization in v0.7.0+.

### Stereo Rendering Pipeline

```
Per frame:
  1. xrWaitFrame() — block until compositor is ready
  2. xrBeginFrame()
  3. xrLocateViews() — get left/right eye poses + FOVs
  4. For each eye:
     a. xrAcquireSwapchainImage()
     b. xrWaitSwapchainImage()
     c. Bind framebuffer to swapchain image
     d. Set viewport (1920x1920)
     e. Compute view + projection matrices from eye pose + FOV
     f. Render scene nodes (surface quads with DMA-BUF textures)
     g. xrReleaseSwapchainImage()
  5. xrEndFrame() — submit both layers
```

### Dependencies

- Phase 2 complete (DRM lease working, Monado has display access)
- `vr_renderer.rs` shaders compile on GLES 3.2 (already written)
- `scene.rs` scene graph operational (already written)
- `texture.rs` DMA-BUF import path (stub exists, needs real Vulkan/GL interop)

### Risks

- **Vulkan/GL interop on AMD** (high): The renderer uses GLES via glow, but
  Monado's Vulkan compositor may require VK_EXT_image_drm_format_modifier for
  texture sharing. May need to switch renderer to pure Vulkan (ash) or use
  EGL_EXT_image_dma_buf_import. **This is the highest-risk item in v0.6.0.**
  Mitigation: prototype texture sharing early in 3.5, fall back to CPU readback
  if interop fails (slow but functional for validation).
- **Frame timing jitter** (medium): X99 platform (PCIe 3.0) may introduce
  latency spikes on DMA-BUF transfers. Mitigation: gpu_power.rs sets
  performance mode during VR sessions; frame timing telemetry will expose issues.
- **Monado session state races** (low): OpenXR session lifecycle has async state
  transitions. `openxr_state.rs` already handles all states but only in stub mode.
  Real runtime may surface edge cases (loss recovery, runtime restart).

| 4.5 | Camera -> MJPEG decode -> pupil center extraction -> gaze vector | L | rust |
| 4.6 | Calibration routine: 9-point gaze calibration with on-screen targets | M | rust+elisp |
| 4.7 | Wire gaze vector into `eye_tracking.rs` as `GazeSource::Bigeye` variant | M | rust |
| 4.8 | Feed gaze data to `gaze_focus.rs` -> focus routing -> ewwm-vr-eye.el | M | rust+elisp |
| 4.9 | Latency measurement: camera-to-focus < 20ms target | M | profiling |
| 4.10 | Add `GazeSource::Bigeye` to IPC status commands | S | rust+elisp |

### Eye Tracking Architecture

```
Bigeye camera (USB, 35bd:0202)
  |
  v4l2 / libuvc
  |
  MJPEG decode (turbojpeg or ffmpeg)
  |
  800x400 grayscale frame
  |
  Pupil detection (per eye: 400x400)
  |  Options:
  |  A. Baballonia ML model (ONNX, runs on GPU via onnxruntime)
  |  B. Traditional CV: threshold -> contour -> ellipse fit (CPU, lower latency)
  |  C. OpenXR XR_EXT_eye_gaze_interaction (if Monado driver supports Bigeye)
  |
  Pupil center (px, py) per eye
  |
  3D gaze ray (calibration-dependent)
  |
  eye_tracking.rs -> gaze_focus.rs -> IPC -> ewwm-vr-eye.el
```

### Bigeye Camera Details

- **Resolution:** 800x400 (400x400 per eye, side-by-side)
- **Format:** MJPEG (compressed), ~90fps
- **Interface:** UVC (USB Video Class) — standard v4l2 should work
- **USB:** High Speed (480Mbps), PID 0x0202
- **Illumination:** IR LEDs (built into headset)

### Dependencies

- Phase 3 complete (VR session running, have a visual target for calibration)
- UVC kernel driver functional for Bigeye
- Pupil detection model or algorithm chosen

### Risks

- **Bigeye firmware/UVC quirks** (high): The Bigeye camera may need a kernel
  patch for `dwMaxVideoFrameSize` validation (known issue with non-standard UVC
  cameras). Mitigation: libuvc bypasses kernel UVC entirely.
- **Calibration accuracy** (medium): Without Bigscreen's cloud calibration
  pipeline, accuracy depends on our calibration routine quality. The Baballonia
  bsb2e_linux fork provides a starting point but is early-stage.
  Mitigation: start with 9-point calibration, iterate on accuracy.
- **GPU contention** (medium): Running pupil detection ML model on the same
  GPU doing VR rendering may cause frame drops. Mitigation: use CPU-based
  ellipse fitting for v0.6.0, defer ML model to v0.7.0; or use async compute
  queue.

```yaml

name: Hardware VR Tests
on:
  push:
    paths: ['compositor/src/vr/**']
  workflow_dispatch:

    runs-on: [self-hosted, honey, vr]

      - uses: actions/checkout@v4
      - run: cargo build --features vr,full-backend
      - run: |
          # Start compositor in headless+VR mode
```

### Dependencies

- Phases 3 (or 4) complete for meaningful profiling data

### Risks

- **CI flakiness** (medium): VR hardware tests are inherently less deterministic
  than unit tests. Frame timing depends on system load, GPU thermals, USB latency.
  Mitigation: use generous timing thresholds (< 15ms instead of < 11.1ms),
  retry flaky tests once.
- **Self-hosted runner security** (low): honey is on a private network.
  Use runner groups and labels to restrict which workflows can target it.

## Key Technical Decisions for v0.6.0

### 1. Rendering API: GLES 3.2 via glow (keep current approach)

`vr_renderer.rs` already uses glow + GLES 3.2 shaders. Monado's compositor
handles Vulkan internally. We render to GL framebuffers, Monado composites to
the Vulkan swapchain. This avoids a full renderer rewrite.

If Vulkan/GL interop fails (risk item in Phase 3), the fallback is to use
`ash` (Vulkan bindings) directly, which is a significant effort increase
(~1-2 additional weeks).

### 2. Eye tracking: Start with ellipse fitting, not ML

For v0.6.0, use traditional computer vision (threshold + contour + ellipse fit)
for pupil detection. This runs on CPU with < 2ms latency and avoids GPU contention.
Defer Baballonia's ONNX model to v0.7.0 when we can profile GPU headroom.

### 3. DRM lease: Smithay native, not raw ioctl

Use Smithay's `drm_lease` module rather than raw `drmModeCreateLease()`.
This integrates cleanly with the existing compositor backend and handles
CRTC/plane allocation correctly.

### 4. Frame timing: Target 90Hz, accept 75Hz initially

The Beyond supports 75Hz and 90Hz. Start at 75Hz (13.3ms budget) for more
headroom during initial bringup. Switch to 90Hz after profiling confirms
we can hit the budget consistently.

   `grep -r "BIG.*0x1234\|bigscreen.*beyond" drivers/gpu/drm/drm_edid.c`

   enough for production, or should we implement UVC capture from scratch?

4. **GL/Vulkan texture sharing path** — Does Monado expose its Vulkan device

   Mode switching requires reflashing JSON firmware via `lighthouse_console`.
